# Jematell performance + motion

## Responsive WebP images
`src/components/ResponsiveImage.tsx` renders a `<picture>` with WebP `srcset`/`sizes`
variants (generated by `scripts/gen-images.mjs` at q90, visually lossless) plus the
original JPEG fallback. `picture { display: contents }` keeps the wrapper
layout-neutral so existing CSS targeting the inner `<img>`/class is unchanged;
intrinsic `width`/`height` give zero CLS. Re-run `node scripts/gen-images.mjs` after
adding/replacing a source JPEG in `public/images/`.

**Hero exception (do not undo):** `public/images/hero.jpg` is itself an
already-compressed 2500px WebP, so re-encoding it to downscaled WebP variants
double-compressed the aerial photo and visibly softened it. The hero renders the
original `hero.jpg` directly (full-res `<img>`, browser-downscaled) and is preloaded
in `index.html` for LCP. Do NOT re-add hero variants to `gen-images.mjs`.

## Per-route code splitting (eager on server, lazy on client)
`src/routes.tsx` splits every non-home page into its own chunk, branching on
`import.meta.env.SSR`: SSG `renderToString` is synchronous and cannot suspend on
`React.lazy`, so the server top-level-awaits the real modules and emits full HTML,
while the client gets `React.lazy` chunks. `<Routes>` is wrapped in `<Suspense>`;
React 18 keeps the server HTML during hydration (no blank flash). Home + NotFound
stay eager. Dropped initial JS from ~1.2MB to ~190KB.

**Rule:** when adding a route, add it to BOTH branches (eager `await import` under
`import.meta.env.SSR`, AND client `React.lazy`). A client-only-lazy route renders an
empty Suspense fallback into the prerendered HTML and loses its SEO content.

## Fonts via non-blocking `<link>`
Google Fonts stylesheet loaded with a `<link>` in `index.html` (preconnect present),
NOT a CSS `@import` (which chain-blocks render). Newsreader + Outfit narrowed to the
weights actually used (300-600, italic kept), `font-display: swap`.

## framer-motion via LazyMotion (and the m-vs-motion traps)
`AppShell` wraps the app in `<LazyMotion features={domAnimation} strict>`, and every
component imports `m` (not `motion`). Ships only the DOM-animation feature set instead
of the full `motion` engine. `domAnimation` is sufficient because nothing uses
`layout`/`drag` (those need `domMax`). `MotionConfig`/`AnimatePresence`/
`useReducedMotion` are unaffected.

- **Use `m`, never the full `motion`.** `strict` mode makes `motion.div` throw at
  runtime. If you ever need `layout`/`drag`, switch the feature set to `domMax`.
- **A bare `motion.*` only fails at runtime, not at build.** Vite/esbuild does NOT
  typecheck, so a component using `motion.div` while importing only `m` builds fine but
  throws during SSG `renderToString`. The prerender then writes a fallback page with no
  `<h1>`/body, silently failing the `audit` content check for ONLY the routes that
  render the broken component. Always run `typecheck` (catches it as TS2552) and the
  `audit` before trusting a build.

## Contact form is lazy-loaded
`ContactFormProvider` loads `ContactForm` via `React.lazy` (Suspense fallback={null},
inside `AnimatePresence`), so the ~47KB form chunk only downloads when first opened.
It is behind a button (no SEO value), so this is safe.
